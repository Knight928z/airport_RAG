机场知识库 RAG 助手｜完整实现全流程细节文档

版本说明
- 文档目标：完整描述当前项目在代码层面的实现路径与运行全链路，便于交接、运维、二次开发。
- 适用范围：当前仓库（airport_RAG）主分支实现。

==================================================
一、项目定位与目标
==================================================
本项目是一个面向机场业务场景的中文 RAG（Retrieval-Augmented Generation）系统，核心目标：
1) 对机场/航司规章进行可追溯问答（有引用依据）
2) 对复杂高风险问题采用规则优先，减少幻觉
3) 提供管理后台（文档CRUD、批量上传、OCR校对、补丁治理）
4) 接入实时航班能力（MCP）并以卡片结构展示
5) 形成“用户反馈 -> 自动补丁 -> 入库 -> 再回答”的闭环

==================================================
二、总体架构
==================================================
系统分为 7 层：

A. 接口层（FastAPI）
- 文件：src/airport_rag/api.py
- 负责：HTTP 路由、请求校验、前后端数据交换、流程编排。

B. 业务编排层（RAG Service）
- 文件：src/airport_rag/service.py
- 负责：检索、重排、规则判定、低置信兜底、模型生成（可选）。

C. 规则层（Rules Engine）
- 文件：src/airport_rag/rules.py
- 负责：关键词体系、跨语言扩展、主题识别、专项规则回答（行李、票务、海关、边检等）。

D. 实时航班层（MCP Client）
- 文件：src/airport_rag/realtime_flight.py
- 负责：MCP工具发现、参数适配、结果标准化、结构化字段抽取。

E. 检索存储层（Chroma + Embedding）
- 文件：src/airport_rag/vector_store.py, embeddings.py
- 负责：向量入库、向量查询、维度异常与查询不稳定兜底。

F. 入库层（Ingest + OCR）
- 文件：src/airport_rag/ingest.py
- 负责：文档读取、切分、元数据生成、图片OCR提取。

G. 前端展示层（静态页面）
- 文件：src/airport_rag/static/index.html（问答页）
- 文件：src/airport_rag/static/admin.html（管理页）
- 文件：src/airport_rag/static/patches.html（补丁治理）
- 文件：src/airport_rag/static/ocr_review.html（OCR校对）

==================================================
五、文档入库全流程
==================================================
入口：
- POST /ingest
- POST /ingest/default
- 管理端上传（/admin/docs, /admin/docs/bulk）

流程细节：
1) ingest.load_documents(path)
   - 支持 .txt/.md/.pdf/无后缀文本
   - 支持图片（png/jpg/jpeg/webp/tiff/bmp）OCR提取
2) split_text（基于 langchain_utils）分块
3) 生成 chunk_id（uuid5，基于 source|page|index|content）
4) 提取元数据
   - source, page, doc_scope, carrier
5) embeddings.embed_documents
   - 优先 sentence-transformers
   - 若加载失败，回退 hashing embedding
6) vector_store.add_chunks(upsert)
   - 处理维度不一致异常（重建 collection）

==================================================
六、问答（/ask）全流程
==================================================
请求模型：AskRequest
- question
- top_k
- enable_realtime（默认 false）

A. 实时航班触发策略（当前实现）
- /ask 默认不自动实时（enable_realtime=false）
- 只有 enable_realtime=true 且问题中含明确航班号，才会尝试实时航班
- 目的是降低误触发

B. 常规RAG流程（service.ask）
1) _ensure_index_ready：索引为空时尝试从 data/documents 预热入库
2) 构建 source policy（机场/航司范围）
3) 跨语言查询扩展（中英互补词）
4) 向量检索 + fallback 查询扩展检索
5) 重排 + 相关性过滤 + 聚焦
6) 规则优先生成：
   - 票务保险退保
   - 文档查找
   - 退改签
   - 高优先规则（如电池、出发时效）
7) 低置信判定（无明确事实）
8) 如配置 OpenAI 可模型生成，否则抽取式回答
9) 生成 citations（source/page/snippet/chunk_id）

C. 低置信处理
- confidence_note in {low-confidence, index-empty}
- 自动写入 uncovered_questions.jsonl（由 /ask 路径触发）

==================================================
七、实时航班（/flight/realtime）全流程
==================================================
入口：POST /flight/realtime
- 输入：question 或 flight_no
- 输出：
  - answer
  - realtime_flight（标准卡片）
  - realtime_flight_details（明细字段）
  - realtime_flight_labels（后端映射后的中文标签）

流程细节：
1) 实时意图识别
   - normalize_flight_no + 关键词判定
2) MCP 客户端初始化
   - VariFlightMCPClient
3) 工具发现/选择
   - 优先 searchFlightsByNumber（有航班号）
4) 参数候选重试
   - 多套 arguments 逐个调用，减少工具参数差异导致失败
5) HTTP 兼容
   - 支持 307/308 重定向
   - Accept 包含 application/json,text/event-stream
6) 结果解析
   - 提取 text
   - 提取结构化 details（含嵌套 dict 文本解析）
   - 构建标准化 RealtimeFlightCard
7) 异常识别
   - 工具 error 文本识别并抛错
8) 自动落盘
   - data/documents/airport/实时航班/YYYY-MM-DD-<航班号>.md
9) 自动入库
   - 落盘后触发 ingest，便于后续检索追溯

==================================================
八、实时字段中文映射实现
==================================================
映射来源：
- data/documents/airport/实时航班/航班字段（Markdown表格）

后端实现：
1) /flight/field-mappings
   - 读取字段映射表并返回 mappings
2) /flight/realtime
   - 根据本次 details.keys() 计算 realtime_flight_labels
   - 使用 canonical key 匹配（兼容 FlightNo / flight_no / flight-no）

前端实现（index.html）：
1) 详情渲染优先使用 realtime_flight_labels
2) 若缺失，再回退到 /flight/field-mappings 加载结果
3) 前端也有 normalize/canonical 匹配兜底

当前展示格式：
- 中文标签：字段值（原始Key）
例：航班号：CZ325 (FlightNo)

==================================================
九、前端问答与“二次确认查询实时航班”
==================================================
页面：/app -> static/index.html

交互策略：
1) 用户点击“开始问答”
2) 前端请求 /ask 时固定 enable_realtime=false
3) 若问题疑似航班实时意图：显示提示
   “系统暂无此航班信息，是否查询？”
4) 用户点击“查询实时航班”后，才调用 /flight/realtime
5) 返回卡片 + 可切换详细字段展示

目标：
- 避免普通问题误触发实时航班API
- 把实时查询从“自动触发”变“用户确认触发”

==================================================
十、反馈闭环与补丁治理
==================================================
入口：POST /feedback

链路：
1) 写 answer_feedback.jsonl
2) 触发 _maybe_apply_feedback_patch
   - 去重（fingerprint）
   - 分层（high/medium/low）
   - 按主题归档
3) 达阈值自动 merge 到 知识补丁合并稿
4) 触发 ingest 同步
5) 尝试 service.ask 生成 iterated_answer

治理接口：
- GET /admin/patches/stats
- POST /admin/patches/review-merge

十五、常见问题与排障
==================================================
1) 问答一直低置信/无引用
- 检查 data/documents 是否有内容
- 调用 /ingest/default
- 检查 data/chroma 是否可写

2) 实时航班字段仍英文
- 检查 data/documents/airport/实时航班/航班字段 是否存在且格式为 |字段名|信息映射|
- 检查 /flight/field-mappings 返回
- 检查 /flight/realtime 返回 realtime_flight_labels

3) 二次确认按钮不出现
- 仅“航班实时意图”问题才显示
- 前端调用 /ask 默认 enable_realtime=false
- 按钮点击后才调用 /flight/realtime

4) embeddings 模型慢或加载失败
- 可先用默认回退 hashing 保障可用性
- 若需高质量向量，准备好本地缓存模型并启用下载策略

5) chromadb 维度错误或查询不稳定
- 已内置 collection 重建和 query fallback
- 可清理 data/chroma 后重建索引
